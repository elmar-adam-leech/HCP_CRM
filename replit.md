# Multi-Tenant CRM System

## Overview
This project is a multi-tenant Customer Relationship Management (CRM) system for service-based businesses, primarily HVAC contractors. It aims to streamline customer management, job tracking, estimates, lead nurturing, and communication. The system ensures data isolation, provides robust role-based access control, allows seamless multi-company user switching, and focuses on efficient business operations and market potential.

## User Preferences
Preferred communication style: Simple, everyday language.
Performance is a first-class requirement — every change is evaluated for impact on load time, payload size, and end-user device resources.

## System Architecture

### UI/UX Decisions
The frontend uses React with TypeScript, Vite, Wouter, and TanStack Query. It features a custom component library built with Radix UI and Tailwind CSS, inspired by Pipedrive's design with shadcn/ui. The design is mobile-first, responsive, uses card-based layouts, and includes light/dark mode theming. Productivity is enhanced via a global command palette and context-aware keyboard shortcuts. All communication actions are standardized through centralized components and hooks. Visual design tokens and rules are defined in DESIGN.md (Google Labs design.md alpha format).

### Technical Implementations
The backend is built with Node.js and Express.js, providing a RESTful API. PostgreSQL, managed with Drizzle ORM, ensures type-safe database operations and multi-tenancy. Data validation uses Zod schemas. Authentication uses JWT tokens, HTTP-only cookies, and bcrypt. The system implements role-based access control and strict tenant data segregation, supporting multi-contractor users with seamless switching. Real-time updates are facilitated by WebSockets. Performance is optimized through database indexing, application caching, and query optimization. Background loops are kept lean to minimize idle Autoscale compute: the former always-on cleanup timers (auth-cache sweep, rate-limit prune, gmail-oauth-state cleanup, refresh-token cleanup, revoked-token cleanup, spam-audit prune, 30-day webhook-events cleanup, and the data-retention check) are consolidated into a single daily maintenance pass (`server/services/maintenance-job.ts`, 3 AM UTC, each step independently try/caught). The sales-process cron and Dialpad event poller are adaptive/self-scheduling (sleep until the next item is due / back off geometrically toward a 5-minute cap while idle) and are nudged back to a fast interval on new work; a cached `hasAnyEnabledIntegration` gate (`server/services/integration-presence.ts`) short-circuits the Dialpad poller and the Dialpad/HCP health checkers when no contractor uses that integration. None of these changes alter user-visible behavior. **Background-job topology (task #802):** every periodic job can run EITHER on the always-on in-app timers (the default — `RUN_IN_APP_JOBS` env var unset or any value other than `"false"`) OR on standalone Replit Scheduled Deployments (cron) so the Autoscale web app can scale to zero when idle. The shared entrypoint is `server/worker.ts`, bundled to `dist/worker.js` by the `build:worker` npm script (esbuild, `--platform=node --packages=external --bundle --format=esm`, mirroring the web-app build; it is also chained onto the main `build`). In production run `NODE_ENV=production node dist/worker.js <job[,job...]>` (the old `npx tsx server/worker.ts <jobs>` still works in dev where tsx is available) — it runs one or more jobs once and exits; available jobs are `sync`, `sales`, `workflows`, `dialpad`, `health`, `ad-spend`, `cleanup`, `maintenance`. Each invocation runs under a Postgres session advisory lock (`server/jobs/job-lock.ts`, `withJobLock(name, fn)` → `pg_try_advisory_lock`) so two overlapping scheduled runs — or a scheduled run overlapping an in-app timer during the migration window — cannot double-process; this composes with each job's existing idempotency (atomic claims, echo suppression). The web app's one-shot job hooks are `syncScheduler.runOnce()`, `workflowEngine.runSuspendedPollOnce()`, `dialpadEventPoller.runOnce()` + `waitForDialpadQueueDrain()`, the sales `runDueAutoTasksOnce()` drain loop, the health checkers, `messageCleanupService.performCleanup()`, `runDailyMaintenance()`, and `runAdSpendSync()`. The worker never runs DDL (the web app still owns schema migrations on boot) and has a `WORKER_JOB_TIMEOUT_MS` watchdog (default 10 min). WebSocket real-time and all live request-driven work (webhook ingestion, manual syncs, workflow triggers) always run in the web app regardless of `RUN_IN_APP_JOBS`. To cut over: in the Publishing UI create Scheduled Deployments (build `npm install && npm run build:worker`, run `NODE_ENV=production node dist/worker.js <jobs>`) at each job's cadence, then set `RUN_IN_APP_JOBS=false` on the web app. Do NOT convert the main web app to a scheduled deployment. Database schema changes are managed via idempotent `columnMigrations` in `server/schema-drift.ts` (NOT `server/db.ts` — that file only calls the drift helpers) to ensure all tenant databases stay in sync; `runSchemaDriftCheck()` fails loudly at boot and in CI (`server/scripts/check-schema-drift.ts`) when a Drizzle-declared column is missing from a database.

### Feature Specifications
- **Multi-Tenancy**: Data isolation and secure access control per tenant.
- **Leads, Estimates & Jobs**: Full CRUD operations with dual-entity architecture (contacts vs. leads) and real-time updates.
- **Communication & Messaging**: SMS integration, unified conversation threads, and automatic activity capture.
- **Gmail Integration**: Per-user OAuth, encrypted token storage, automatic inbox syncing, and email activity capture with intelligent reply matching. Includes a shared company email feature.
- **Unified Scheduling System**: Integration with external platforms, unified calendar views, appointment slot enforcement, and auto-assignment.
- **Workflow Automation Builder**: Visual, drag-and-drop builder using React Flow, supporting custom nodes, dynamic variables, and execution logs.
- **Facebook Leads Integration**: Automatic ingestion of leads from Facebook Lead Ads via OAuth and webhooks, with a polling fallback.
- **Dialpad Integration**: Handles calling, SMS, and call recording, including company numbers management and event ingestion.
- **Twilio Integration**: First-class calling/texting provider alongside Dialpad (module lives in `server/twilio/*`). Contractor-scoped credentials only (no system-wide fallback); raw Twilio REST API over HTTP Basic auth (no SDK). Syncs IncomingPhoneNumbers into `twilio_phone_numbers`. Outbound SMS/calls resolve their provider server-side via `server/providers/provider-service.ts` (tenant `tenant_providers` preference + a UI "From Number" picker aggregating both integrations' numbers). Outbound calls use a **bridge model** — Twilio first rings the rep's phone (their per-user `user_contractors.twilio_phone_to_ring`), then the `/api/webhooks/twilio/voice/bridge/:tenantId` TwiML `<Dial>`s the customer. Inbound calls run a configurable **ring tree** (`server/twilio/ring-tree.ts`) — ordered steps, each ringing its members simultaneously, falling through to voicemail. Call recordings are streamed through an authenticated proxy (`server/twilio/recordings.ts`). **Inbound-SMS routing (task-critical):** for numbers inside a Twilio Messaging Service (A2P 10DLC), Twilio ignores the number-level `SmsUrl` and uses the Service's own inbound setting, so `server/twilio/webhook-config.ts` sets the Service `InboundRequestUrl` DIRECTLY to our tenant-scoped SMS webhook (with `UseInboundWebhookOnNumber=false`) rather than relying on number-level deferral. Webhook receivers (`/api/webhooks/twilio/*`) verify the `X-Twilio-Signature` HMAC-SHA1 against the exact posted URL before processing.
- **HCP (Housecall Pro) Integration**: Full bi-directional sync for contacts, estimates, and jobs with echo suppression and two-tier webhook authentication. Includes booking and calendar sync.
- **Lead Capture / Public Booking**: Public-facing booking page for estimates, including HCP calendar sync and structured address capture. Booker-typed notes are pre-staged via `POST /leads/{leadId}/notes` before convertLead so HCP carries them into the resulting estimate, with a verification re-fetch and bounded retry; on failure the booker sees a warning banner and contractor admins receive a SendGrid email. Fix is forward-looking only — historical bookings (e.g. pre-fix records like Meagan Petri) are not backfilled.
- **Google Maps Address Autocomplete**: Integrates Google Places API for address autocomplete.
- **Reports**: Provides reports like Leads Trend and "Speed to Lead by Salesperson".
- **AI Scheduling Agent (settings only — task #697 agent itself still being built)**: Per-contractor admin-only settings on the Account tab to (a) toggle the AI SMS scheduling agent on/off, (b) edit a free-text "Personality & Tone" prompt, and (c) edit a "Company Context" prompt. Stored on `contractors` (`ai_scheduling_enabled`, `ai_scheduling_personality`, `ai_scheduling_company_context`) and managed via `GET`/`PATCH /api/settings/ai-scheduling` (admin-only). UI lives in `client/src/components/settings/account/AiSchedulingAgentCard.tsx`. The runtime agent that consumes these values is built separately under task #697.
- **Real-time**: WebSocket-based architecture with reconnect/stale-data banner.
- **Security**: HTTP-only cookies, role-based access control, AES-256-GCM encryption, JWT revocation, `tokenVersion` for sign-out-all-devices. Task #737 adds a cookieless bearer-token fallback so installed PWAs (iOS Safari) stay signed in when the `auth_token` cookie AND IndexedDB are evicted together: the auth JWT is mirrored into BOTH `localStorage["auth-token"]` AND IDB `auth-fallback/auth_token` (newer-write wins on read), the refresh token is similarly mirrored into BOTH `localStorage["refresh-token"]` AND IDB `auth-fallback/refresh_token`, and `apiRequest`/`getQueryFn` attach `Authorization: Bearer <jwt>` on every request when a stored token is present (cookies still go via `credentials: "include"`; the server prefers the cookie when both arrive). Every mint endpoint (`POST /api/auth/login`, `POST /api/auth/mfa/verify`, `POST /api/auth/webauthn/login/finish`, and BOTH branches of `POST /api/auth/refresh` — rotated and grace) returns the raw auth JWT in the JSON body as `authToken` (matching the `auth_token` cookie value, pinned by `auth-mint-contract.test.ts` / `webauthn.test.ts` / `auth-refresh.test.ts`). The refresh response also includes `via: "bearer" | "cookie"` so the SPA can pin the working delivery path for the rest of the tab. Dead-token outcomes (`not-found` / `revoked` / `expired` / `replayed-past-grace` / `membership-missing`) clear BOTH stores, and every logout path funnels through the single `clearAllStoredAuthTokens()` helper to avoid drift. A 1%-sampled `auth_source_sample` log (`AuthService`) tracks cookie-vs-bearer usage in production. Storing the raw JWT client-side is strictly weaker than the httpOnly cookie (an XSS would expose it) — the cookie remains the default and bearer is a fallback only; see `threat_model.md` PWA storage trade-off for the rationale. Persistent PWA login uses a 90-day `refresh_token` httpOnly cookie alongside the 7-day `auth_token`; the SPA silently re-mints the auth token via `POST /api/auth/refresh` (single in-flight, refresh rotates the row) when any `/api/*` call returns 401, so iOS Safari evicting the auth cookie no longer logs the user out. The refresh endpoint enforces a 30-second rotation grace window — re-arrivals of an already-rotated token within that window succeed without re-rotating (covers in-flight retries / network re-sends), while reuse past the window is treated as a replay attack and hard-revokes the row. Two stacked rate limiters guard the endpoint: 10/min per-IP and 5/min per-refresh-token (so a leaked cookie cannot be brute-forced from a botnet). Task #720 adds an IndexedDB-backed fallback: login/MFA/passkey responses now mirror the raw refresh token in the body, the SPA persists it to `auth-fallback`/`tokens`, and `/api/auth/refresh` accepts the same token via `body.token` when iOS Safari evicts the cookie outright. Both delivery paths run through identical rotation/replay/rate-limit checks; the IDB copy is rotated on every successful refresh and cleared by both logout call sites (`Header`, `SecurityTab`) and by 401 reasons that prove the stored token is dead (`not-found`, `revoked`, `expired`, `replayed-past-grace`, `membership-missing`). The refresh endpoint emits structured per-outcome telemetry via `logger('AuthRefresh')`. **Response contract (enforced by tests in `server/auth-refresh.test.ts` and pinned by #734):** every endpoint that mints an `auth_token` cookie — `POST /api/auth/login`, `POST /api/auth/mfa/verify`, `POST /api/auth/webauthn/login/finish`, and the rotated branch of `POST /api/auth/refresh` (both the cookie source and the body-only/IDB-fallback source) — MUST also include a non-empty `refreshToken` string in the JSON body whose value matches the raw value placed in the `refresh_token` cookie. The grace branch of `/api/auth/refresh` returns `{ ok: true, grace: true }` without a `refreshToken` because no rotation occurred and the existing IDB copy is still authoritative. Task #734 hardens the persistence path: the SPA now uses `setStoredRefreshTokenStrict` (which surfaces IDB errors instead of swallowing them), and on failure dispatches an `auth-persist-failed` window event and POSTs `{stage,errorName}` to `/api/auth/persist-failed` (no token, no PII, 5/min/IP, logged via `logger('AuthPersist')`). Refresh-attempt telemetry now also includes `bodyTokenPresentSource: "cookie" | "idb-fallback" | "none"` so empty-IDB outcomes are distinguishable from cookie-only and IDB-fallback paths in production logs.
- **PWA passkey-first cold-start (task #738)**: Installed PWAs that have a registered passkey resolve their cold-start authentication along the chain `cookie → bearer → passkey-conditional → passkey-explicit → password`. The `LoginForm` email input carries `autoComplete="username webauthn"` and (when `hasLocalPasskey` is set + the device has a platform authenticator) opens a long-lived `attemptConditionalPasskey()` request via `client/src/lib/boot-auth.ts` so iOS Safari / Chrome show Face ID right inside the autofill prompt — successful conditional flows persist tokens and report `bootResolution: "passkey-conditional"` to `/api/auth/storage-probe`. When a local passkey is registered the password fields are collapsed behind a "Sign in another way" toggle so Face ID is the only visible CTA; users without a local passkey see the password form unchanged. The boot-time helper `attemptBootSilentPasskey()` runs BEFORE React mounts: it tries conditional mediation first (reports `passkey-conditional`) and, if conditional is unavailable on the platform, falls back to the explicit/system-picker flow (reports `passkey-explicit`). The user-initiated "Sign in with Face ID" button in `AppInner.handlePasskeyLogin` also reports `passkey-explicit`; the password sign-in path reports `password`. The pkhint=1 cookie is a priority signal only — boot still attempts the unlock when the cookie was wiped alongside the auth cookies (the exact iOS storage-partition failure mode this task targets), and the OS itself is the source of truth for whether a discoverable credential exists. The same helper exposes `determineBootSource()` (cookie/bearer/none) which `main.tsx` reports to `bearer_probe` so production telemetry can pivot fallback usage by outcome. A new public, rate-limited (10/min/IP) endpoint `GET /api/auth/webauthn/has-credentials` lets the client ask "is a passkey worth trying?" without becoming an enumeration oracle: for any well-formed `?email=` it returns `{ hasAny: true }` unconditionally (the DB is never consulted on that branch), for a malformed email returns `{ hasAny: false }`, and without an email gates on the non-secret `pkhint=1` cookie that `PasskeysCard` writes alongside `localStorage["hcp.webauthn.hasPasskey"]` after a successful registration. After a password-based first login, eligible users (zero passkeys, never dismissed, platform authenticator available) are shown `PasskeyEnrollmentPrompt` once — both "Set up Face ID" and "Maybe later" call `POST /api/auth/passkey-prompt/dismiss` which sets `users.passkey_prompt_dismissed_at`, surfaced through `/api/auth/me` (along with `passkeyCount` from `webauthn_credentials`) so the dialog never re-appears uninvited. Pinned by `server/routes/has-credentials.test.ts` (no-enumeration + 10/min/IP rate limit) and `client/src/lib/boot-auth.test.ts` (boot-source resolution + telemetry never-throws).
